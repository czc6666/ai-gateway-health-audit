# AI API 网关健康审计｜脱敏样例

> 本样例仅展示交付结构，不对应任何客户环境。

## 1. 结论

当前链路可以处理请求，但尚不能证明“可安全重启、路由可解释、统计可恢复、故障可回滚”。

## 2. 风险摘要

| 等级 | 发现 | 证据 | 建议 |
|---|---|---|---|
| P0 | 重启恢复脚本可能用陈旧快照覆盖最新用量 | 启动顺序与恢复文件时间戳不一致 | 重启前导出 live usage；更新恢复快照后影子验证 |
| P1 | API-key 账号使用仅 OAuth 生效的模型映射 | 配置语义与真实 `/v1/models` 不一致 | 客户端改用原生模型名，不依赖无效映射 |
| P1 | 健康检查只验证进程存活 | 进程正常时真实请求仍可 503 | 增加最小真实 completion 与上游错误分类 |
| P2 | 账号池缺少失效账号隔离口径 | disabled/expired 与 active 混合统计 | 报表分离 live、expired、free、disabled |

## 3. 建议执行顺序

1. 只读导出当前配置、路由和 live usage。
2. 验证 `/v1/models` 与最小 completion，不以进程状态代替链路健康。
3. 在独立端口启动影子实例，回放脱敏测试矩阵。
4. 确认统计持久化与恢复快照一致。
5. 小流量切换；旧端口保留为回滚出口。

## 4. 回滚条件

- 错误率高于原链路；
- P95 延迟显著恶化；
- 关键模型缺失或路由到错误上游；
- 重启后使用统计倒退；
- 无法在 5 分钟内解释 401/429/503 的来源层级。

## 5. 无法证明

未提供脱敏重启日志与统计快照时间线，因此不能证明当前恢复机制一定会覆盖总账；只能判定为高风险待验证项。
